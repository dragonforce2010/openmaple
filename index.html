Control plane first. Execution path second.

OpenMaple separates managed-agent resources from the runtime and sandbox providers that execute them. This keeps disaster recovery, cloud migration, and agent-loop experiments practical.

01

Control Plane

Workspace truth, agents, sessions, vaults, model configs, members, API keys, and audit-ready event records.

  • Tenant-scoped access
  • Workspace API keys
  • Session event timeline
02

Runtime Plane

Agent loops run behind provider adapters: veFaaS today, AWS Lambda / Alibaba FC / local runners tomorrow.

  • Runtime pools
  • Background provisioning
  • Loop-level health
03

Sandbox Plane

Tool execution stays isolated from control data. Swap E2B, veFaaS Sandbox, Vercel, or Docker without rewriting agents.

  • Tool calls
  • Artifacts
  • Credential references

Provider identity unlocks multi-cloud agents.

Tenants configure cloud identities once, then runtime, sandbox, and storage selections become provider plus focused options. This is the path toward failover and workload placement.

cloudProviderIdentity:
  provider: volcengine | aliyun | aws | gcp
  auth: aksk | oidc | role

runtimeProvider:
  type: vefaas | fc | lambda | local

sandboxProvider:
  type: vefaas_sandbox | e2b | vercel | docker

storageProvider:
  type: tos | oss | s3 | gcs
