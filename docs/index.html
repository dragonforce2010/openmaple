OpenMaple 文档

OpenMaple 把智能体从 demo 变成可托管、可观测、可迁移的云软件：控制面记录资源真相，运行面执行 agent loop，沙箱面隔离工具调用。

核心原则

Runtime、Sandbox、Storage、Model 和 Agent Loop 都必须通过 provider 抽象接入，租户只配置云厂商 identity，工作区选择 provider 加少量配置。

OpenMaple Docs

OpenMaple turns agent prototypes into managed, observable, and cloud-portable software. The control plane owns resource truth, the runtime plane runs loops, and the sandbox plane isolates tools.

Core rule

Runtime, sandbox, storage, model, and agent-loop choices enter through provider adapters. Tenants configure cloud identity once; workspaces select provider plus focused options.

架构Architecture

Control Plane

管理 tenants、workspaces、agents、sessions、vaults、models、members、API keys 和事件真相。Owns tenants, workspaces, agents, sessions, vaults, models, members, API keys, and event truth.

Runtime Plane

通过 runtime pool 调度 agent loop，支持 veFaaS、AWS Lambda、阿里 FC、本地 runner。Schedules agent loops through runtime pools: veFaaS, AWS Lambda, Alibaba FC, or local runners.

Sandbox Plane

执行工具调用、文件读写、命令和 artifacts，目标是 E2B、veFaaS Sandbox、Vercel、Docker 可替换。Executes tools, file operations, commands, and artifacts across E2B, veFaaS Sandbox, Vercel, or Docker.

Event Truth

session_events 和 tool_calls 是调试、审计、Ask Maple 与 E2E 验收的共同事实源。session_events and tool_calls are the shared source for debugging, audit, Ask Maple, and E2E checks.

云厂商 identity 与 provider 抽象Cloud identity and provider abstraction

租户侧配置云厂商 identity，例如火山、阿里云、AWS、GCP 的 AK/SK、OIDC 或 role。之后 runtime、sandbox、storage 选择页只需要选择 provider 与关键配置。

Tenants configure cloud identity for Volcengine, Alibaba Cloud, AWS, or GCP through AK/SK, OIDC, or role credentials. Runtime, sandbox, and storage pages then select provider plus focused settings.

Maple CLI

CLI 面向 repo-native 工作流：开发者在本地仓库创建 session、上传上下文、调用 workspace agent，并把 artifacts 拉回本地。

The CLI targets repo-native workflows: create sessions from a local checkout, upload context, invoke workspace agents, and pull artifacts back.

npm install -g maple-agent-cli
maple auth login --workspace openmaple
maple agent list
maple session create --agent codex --env default
maple session send --message "scan this repo and propose fixes"
maple artifacts pull --session sess_xxx
Command用途Use
maple auth配置 workspace API key、server URL、当前 workspace。Configures workspace API key, server URL, and active workspace.
maple agent列出、查看、运行 agent，检查 loop 与模型配置。Lists, inspects, and runs agents with loop/model settings.
maple session创建会话、发送消息、跟踪事件、导出 transcript。Creates sessions, sends messages, follows events, exports transcripts.
maple provider查看 runtime、sandbox、storage provider 与健康状态。Inspects runtime, sandbox, and storage providers with health.

Node / TypeScript SDK

SDK 用于把 OpenMaple 接进后台服务、内部工具、CI 或自定义控制台。认证使用 workspace API key。

The SDK integrates OpenMaple with backend services, internal tools, CI, or custom consoles. Authentication uses workspace API keys.

import { MapleClient } from "maple-agent-sdk";

const maple = new MapleClient({
  baseUrl: process.env.MAPLE_API_URL,
  apiKey: process.env.MAPLE_WORKSPACE_API_KEY,
});

const session = await maple.sessions.create({ agentId, environmentId });
await maple.sessions.sendMessage(session.id, { content: "run the codex path" });
for await (const event of maple.sessions.streamEvents(session.id)) {
  console.log(event.type, event.payload);
}

Skills

Skills 是可复用工作流单元。参考 lark-cli 的拆分方式，OpenMaple 应该把大 skill 拆成职责清晰的子模块，避免一个 skill 同时负责 provider、CLI、部署和排障。

Skills are reusable workflow units. Following the lark-cli style, OpenMaple skills should be split into focused submodules instead of one broad skill that owns providers, CLI, deployment, and troubleshooting.

会话与观测Sessions and observability

Session 是用户消息、agent loop、tool call、sandbox、artifact 的运行边界。列表页用于状态扫描，详情页用于事件审计和失败定位。

A session is the runtime boundary for user messages, agent loops, tool calls, sandboxes, and artifacts. Lists support status scanning; details support audit and failure diagnosis.

  • 失败先看 session.status_failed 的 reason 和 error，再看最近 tool_calls。For failures, inspect session.status_failed reason/error first, then recent tool_calls.
  • Ask Maple 应读取 session_events、tool_calls、artifacts 和 sandbox metadata。Ask Maple should read session_events, tool_calls, artifacts, and sandbox metadata.
  • Codex 链路重点验证：消息发送、事件落库、工具调用、artifact、final answer。Codex path checks: message send, event persistence, tool calls, artifacts, final answer.

Vaults 与 MCP credentialsVaults and MCP credentials

Vault 保存 credential 记录，secret_ref 指向加密密文。Agent 与 session 只引用 credential，不直接暴露密钥。MCP OAuth 需要展示 provider、scope、状态、更新时间和最后使用时间。

Vaults store credential records while secret_ref points to encrypted secrets. Agents and sessions reference credentials without exposing raw secrets. MCP OAuth views should show provider, scopes, status, updated time, and last used time.

发布与 GitHub PagesDeploy and GitHub Pages

OpenMaple 有三类发布：官网文档发布到 GitHub Pages，平台控制台发布到 stable veFaaS，Agent 项目通过 CLI/API 发布成 deployment。三条链路都必须返回可追踪的 resource id，并用 session events、tool_calls、artifacts 作为验收证据。

OpenMaple has three deployment paths: site/docs to GitHub Pages, the console/API to stable veFaaS, and agent projects to deployment resources through CLI/API. Every path must return traceable resource ids and use session events, tool_calls, and artifacts as acceptance evidence.

01GitHub Pages

发布官网、文档、README 图和截图资源。Publishes the site, docs, README images, and screenshot assets.

02Stable veFaaS

更新真实控制台前端、后端 API、网关路由和健康探针。Updates the real console frontend, backend API, gateway routes, and health probes.

03Agent Deployment

把 repo-native Agent 项目发布成可运行、可暂停、可调用的 deployment。Publishes a repo-native agent project as a runnable, pausable, invokable deployment.

发布官网和文档Publish site and docs

部署平台到 stable veFaaSDeploy the platform to stable veFaaS

stable 部署脚本复用已记录的函数和网关资源，只更新前端/后端包，不创建新生产资源。状态文件默认在 output/vefaas/stable-deployment.json，缺失时不要直接 deploy，应先完成 bootstrap 或恢复状态文件。

The stable deploy script reuses recorded functions and gateway resources. It updates frontend/backend packages without creating new production resources. The default state file is output/vefaas/stable-deployment.json; if it is missing, bootstrap or restore the state before deploy.

用 CLI 发布 Agent 项目Deploy an agent project with CLI

maple init/build/deploy 是 repo-native 发布路径。deploy 成功必须返回 deployment_id、agent_id、environment_id，随后用 maple invoke 创建 session 并流式读取事件。

maple init/build/deploy is the repo-native path. A successful deploy returns deployment_id, agent_id, and environment_id; then maple invoke creates a session and streams events.

maple config set api.baseUrl "$MAPLE_API_BASE_URL"
maple config login --api-key "$MAPLE_WORKSPACE_API_KEY"

maple init \
  --name repo-auditor \
  --loop codex_open_source \
  --runtime e2b \
  --directory ./repo-auditor \
  --yes

maple build --project ./repo-auditor
maple deploy --project ./repo-auditor --json
maple invoke "Scan this repo and return evidence." --deployment <deployment_id> --stream
API行为Behavior
GET /v1/deploymentsList deployments visible to the current workspace or API key scope.
POST /v1/deploymentsCreate a deployment from agent, environment, manifest, schedule, and initial events.
POST /v1/deployments/:deploymentId/runRun the deployment with its saved initial event template.
POST /v1/deployments/:deploymentId/invokeInvoke with a user message, create a session, and stream or inspect resulting events.

POST /v1/deployments/:deploymentId/archiveArchive a deployment after preserving its sessions, runs, and audit trail.

Skill 一条命令部署并运行Skill deploy-run in one command

上游 Agent 不应该手写一堆 API 调用。使用 maple skill deploy-run 可以创建 skill、初始化项目、部署 Agent、运行 session，并把 skill_id、deployment_id、session_id、tool_calls 一起返回。

Upstream agents should not hand-roll every API call. maple skill deploy-run creates the skill, initializes the project, deploys the agent, runs the session, and returns skill_id, deployment_id, session_id, and tool_calls.

maple skill deploy-run \
  --name repo-auditor \
  --description "Use when auditing a repository through OpenMaple." \
  --project ./repo-auditor-agent \
  --loop codex_open_source \
  --runtime e2b \
  --prompt "Audit this repository and create a findings summary." \
  --json

部署后验收与失败定位Post-deploy acceptance and failure triage

  • 控制台验收：列表页有 loading，deployment detail 可打开，运行按钮可创建 run/session。Console acceptance: list pages show loading, deployment detail opens, and the run button creates a run/session.
  • 事件验收：session 至少包含 user.message、agent loop event、tool call 或 final answer。Event acceptance: a session includes at least user.message, an agent-loop event, a tool call or final answer.
  • 失败验收：保留 session.status_failed 的 reason 和 error，例如 agent_loop_failed 或 veFaaS runtime error 500，再回查 runtime 日志和 tool_calls。Failure acceptance: preserve session.status_failed reason and error, such as agent_loop_failed or veFaaS runtime error 500, then inspect runtime logs and tool_calls.
  • Codex 链路验收：确认 agent_loop.type=codex_open_source 出现在 deployment manifest，且 session 能写事件、跑工具、产出 artifact。Codex path acceptance: confirm agent_loop.type=codex_open_source appears in the deployment manifest and the session records events, tool calls, and artifacts.
